# 新人育成タスクリスト ── local-ai-project 自走ロードマップ

> **作成**: PM/PMO/PL
> **対象**: local-ai-project にアサインされた新メンバー
> **最終ゴール**: OJT・伴走なしで、新規要件を設計〜実装〜検証〜引き継ぎまで一人で回せる（＝自走）。
> **参照**: `local-ai-project_knowledge.md`（統合ナレッジ。以下「本編」と呼ぶ）。各タスクは本編の該当章とレベル定義（7.5節）に対応。
> **進め方の鉄則**: ①一度に一つずつ変える ②推測より計測 ③やったことはナレッジ化 ④各レベルの合格基準を満たしてから次へ。

---

## 使い方

- 上から順に進める。各タスクに **合格基準** がある。満たせたらチェック。
- 詰まったら、まず本編 第9章（つまずき早見表）と付録B（標準デバッグプロトコル）を自分で当たる。**人に聞く前に切り分けを試みる**のがこの育成の主眼。
- 各レベルの最後に **確認会（レビュー）** を置く。ここだけはPM/PL/先輩が同席。合格したら次レベルへ。
- すべてのタスクは、特記なき限り **Wi-Fiオフ（完全オフライン）** で行う。

## レベル1: 環境を再現できる（模倣フェーズ）

**狙い**: まず「動く状態」を自分の手で再現し、システムの全体像を体で掴む。理解より再現を優先してよい。

- [ ] **1-1** 本編 第0章を読み、環境サマリ（端末・モデル・ディレクトリ構造）を自分の言葉で1枚に図示する。
  - 合格基準: `data/`・`src/`・Ollama・Zed の関係を、見ずに説明できる。
- [ ] **1-2** `ollama list` を実行し、必要モデル（`qwen3.5-9b-16k:latest` / `bge-m3` / OCR・リランカー系）の有無を確認。無ければ pull。
  - 合格基準: 一覧のモデル名を**表記まで正確に**書き写せる（`:latest` の有無まで）。
- [ ] **1-3** 本編 第8章チェックリストに沿って、`src` へ `cd` し、`DATA_DIR` と実際の `data` 位置（`../data`）が一致していることを `pwd`／`ls ..` で確認。
  - 合格基準: なぜ `./data` ではなく `../data` なのかを説明できる。
- [ ] **1-4** Wi-Fiをオフにし、`app_v7_ocr.py` を起動して画面を出す。
  - 合格基準: タイトル・ファイル選択・入力欄が表示される。
- [ ] **1-5** 4つの検証質問（本編 6-3）を投げ、結果を再現する。
  - 合格基準: OCR資料に正答／複合質問で中日本拠地が出る／無い情報に「記載なし」が返る。
- [ ] **1-6** ここまでの手順を、自分用の再現メモとして書き出す（詰まった箇所も記録）。
  - 合格基準: そのメモだけで、翌日ゼロから再起動できる。

**▼ レベル1 確認会（PM/PL同席）**
- [ ] 環境の全体像を口頭で説明 → 3つの検証を実演 → 合格なら レベル2 へ。

## レベル2: 詰まりを自力で切り分けられる（デバッグ自走フェーズ）

**狙い**: このプロジェクトの本質＝「推測でなく事実で真因を名指しする」を身につける。**わざと壊して、自力で直す**のが中心。

> 先輩は、下記の「仕込み」を新人の環境（またはコピー）に事前に入れておく。新人はそれを切り分けで解く。

- [ ] **2-1【仕込み: パスずれ】** `DATA_DIR` を `./data` に書き換えられた状態から、「データフォルダが見つかりません」を自力で解決する。
  - 合格基準: `pwd`／`ls` で場所を確認し、推測でコードをいじる前に事実を取ってから直せる（本編 6-2 詰まり③）。
- [ ] **2-2【仕込み: モデル名違い】** `EMBED_MODEL` や `LLM_MODEL` を存在しない名前にされた状態から、`ollama list` と突き合わせて直す。
  - 合格基準: エラーの出方から「モデル名不一致」を疑い、一覧と照合して修正できる。
- [ ] **2-3【仕込み: 真っ白】** UI描画コードが関数内に埋もれる等でブラウザが真っ白な状態を、F12のConsole/Networkで切り分ける。
  - 合格基準: 「通信は200で正常＝通信は無実」まで事実で確認し、原因を描画側／構造側に絞れる（本編 6-2 詰まり①）。
- [ ] **2-4【裸で計測】** `test_ocr.py` 型の最小スクリプトを**自分で書き**、OCR単体の実処理時間を測る。
  - 合格基準: 「体感フリーズ」と「実処理時間」は別物だと、数字で説明できる（本編 6-2 詰まり②）。
- [ ] **2-5【言語化】** 上記4つの詰まりについて「誤った推測 → 事実確認 → 真因 → 対処」の形で1枚にまとめる。
  - 合格基準: 4件すべてが同じ構造（最初の推測が外れ、切り分けで真因に到達）だと気づき、明文化できる。

**▼ レベル2 確認会（PM/PL同席）**
- [ ] 仕込み4件のうち任意2件をその場で再現・解決 → 切り分け手順を説明 → 合格なら レベル3 へ。
- 評価の主眼は「直せたか」より **「聞く前に切り分けたか」**。

## レベル3: 既存機能を改修・拡張できる（実装自走フェーズ）

**狙い**: トレードオフを理解し、方針判断を含めて既存コードを安全に改修する。本編 第7章の残課題が教材。

- [ ] **3-1【設計先行】** 本編 第7章から課題を1つ選び、着手前に「現状の事実確認 → 改善案 → 想定リスク」を書き出してレビューを受ける。
  - 合格基準: コードを書く前に、何をどう変えるとどのトレードオフが動くかを説明できる。
- [ ] **3-2【Force OCR最適化 / 推奨課題】** 「化け検知したファイルだけOCR、正常ファイルは通常抽出」のハイブリッドに戻す。検知条件は `🄲` 1文字でなく、実際の化け方を踏まえた強化版にする。
  - 合格基準: 正常PDFがOCRを飛ばして高速化し、かつ `travel_rule.pdf` は正しくOCRされることを、両方**事実で**確認（本編 7-1）。
- [ ] **3-3【一度に一つ】** 改修は必ず1変更ずつ行い、その都度オフラインで検証する。
  - 合格基準: 複数を同時変更していない。各変更の前後で挙動差を説明できる。
- [ ] **3-4【版管理】** データやチャンク方式を変えたら `.chromadb_data` を削除し `COLLECTION_NAME` の版数を上げる。
  - 合格基準: 「同名コレクションが古い箱を掴む」現象を理解し、事故を防げる（本編 第8章-7）。
- [ ] **3-5【ナレッジ化】** 実装した改修を、本編と同じ「詰まり→原因→対処」形式で追記する。
  - 合格基準: 追記だけを読んだ第三者が、同じ改修を再現できる。
- [ ] **3-6【Git】** 変更を意味の分かるコミットメッセージで push する。
  - 合格基準: 1コミット＝1つの論理変更になっている。

**▼ レベル3 確認会（PM/PL同席）**
- [ ] 選んだ課題の Before/After を実演 → トレードオフと検証方法を説明 → 追記したナレッジをレビュー → 合格なら レベル4 へ。

## レベル4: 新規要件をゼロから形にできる（設計自走＝ゴール）

**狙い**: 未知の要件を、伴走なしで要件定義〜検証〜引き継ぎまで完遂する。ここを越えたら「自走」達成。

- [ ] **4-1【要件受領】** PMから小さな新規要件を1つ受け取る（例: 「新しい資料種別/フォーマットを追加」「特定部署の資料だけ絞るフィルタを追加」など）。
  - 合格基準: 曖昧な依頼を、達成条件つきの要件に自分で言い換えられる。
- [ ] **4-2【設計】** 実装前に設計メモ（何を作る／どの章の仕組みを流用／リスク）を書き、レビューを受ける。
  - 合格基準: 既存資産（第3・4・5章）の再利用ポイントを自分で特定できる。
- [ ] **4-3【実装・検証】** 一度に一つずつ実装し、オフラインで検証する。
  - 合格基準: 全達成条件を満たすことを事実で示せる。
- [ ] **4-4【方針判断】** 「コードでなく方針を決める」論点（例: データ矛盾をどう扱うか＝本編 7-2）に当たったら、選択肢を整理してPMに意思決定を仰ぐ形にできる。
  - 合格基準: 自分で抱え込まず、選択肢と推奨を提示して判断を引き出せる。
- [ ] **4-5【引き継ぎ】** 本編と同形式のナレッジ（経緯・成果・詰まり・残課題・再現手順）を成果物として納品する。
  - 合格基準: そのドキュメントだけで、別の人が続きを引き継げる。

**▼ レベル4 確認会 ＝ 自走認定（PM/PMO/PL同席）**
- [ ] 新規要件の納品物一式（動くもの＋ナレッジ）をレビュー。
- [ ] 「未知の詰まりに、聞かずに切り分けで到達できたか」を確認。
- [ ] 合格 → **OJT卒業・自走認定**。以降は独立してタスクを持つ。

## 付録: つまずいたときの動き方（新人向け・掲示用）

聞く前に、この順で自分で当たること。これができるのが自走の中身。

1. **本編 第9章（つまずき早見表）** で症状から原因を引く。
2. **推測を疑う**。「重い／壊れている」と思ったら、まず**裸の最小スクリプトで測る**か**ログを見る**。
3. **一度に一つだけ**変えて、変化を確かめる。
4. それでも分からなければ、**「何を試して、何が事実として分かったか」をセットで**先輩に相談する（「動きません」だけはNG）。

> 本編 付録B（実務Tips＝標準デバッグプロトコル）を毎回開くこと。要は3つ:
> **① 推測を疑い、まず測る ② 一度に一つだけ変える ③ 沈黙は原因でなく、見えていないだけ（ログを仕込む）。**
