# 第1章：完全オフライン稼働実験

## 1-1. 実験環境とステータス

- ネットワーク状態：**完全オフライン**（Wi-Fi OFF）
- 使用モデル：`qwen3.5-9b-16k:latest`（Ollama）
- アプリ仕様：Python（Tkinter）＋ ストリーミング出力（`stream: True`）
- 結果：スピナーを突破し、**完全オフラインでのテキスト生成に成功**。ただし次の2課題が浮き彫りになった。
  - 「初速の思考遅延」
  - 「ハルシネーション（盛大な嘘）」

▶補足（再現用）：完全オフライン成功の判定は「Wi-Fiを切った状態でアプリからOllamaへ`http://localhost:11434`経由でリクエストが通り、ストリーミングで文字が返ること」。`localhost` は数字で `127.0.0.1`。エラーログに出ていた `host='127.0.0'` は末尾 `.1` が欠けた不正な文字列で、接続先指定のミスがあるとこの形でタイムアウトする（後述のSystemログ参照）。

## 1-2. 【重要インプット】直面した課題の技術的要因

### 課題①：最初の1文字目が出るまでの「思考時間（沈黙）」が長い

- **原因A（モデルの設計）**：`Qwen3.5` は論理的思考力を高めるため、裏側で「どう回答を組み立てるか」を熟考するステップが挟まる。そのため初速にタメが発生する。
- **原因B（文脈読み込みコスト）**：`16k / 64k` などの大きなコンテキスト長を持つモデルは、会話が長くなるほど「過去の履歴すべて」を毎回最初から巨大な行列計算で再スキャンする。結果、初速の遅延が雪だるま式に増大する。

### 課題②：NPB球団名に対して「ハーレーズ・ホークス」等のデタラメ（嘘）を出力した

- **原因A（パラメータの限界）**：9B（90億）クラスの軽量モデルは、日本のプロ野球のような「ローカルかつ細かい固有名詞」の正確な記憶を保持しきれず、知識の引き出しがバグを起こす。
- **原因B（アプリ側の文脈崩壊）**：通信エラーやタイムアウトでAIの出力が一度「空文字（無言）」になった際、その異常状態も会話履歴（`conversation_history`）に蓄積され、AIの脳内文脈がパニックを起こした。

## 1-3. 実際に起きたこと（生ログ：ハルシネーションの証拠記録）

# 第2章：解決策アプローチ（スペシャリストの対策）

## 対策A：思考スピード（初速）を限界まで高める ―― モデル変更ルート

- **手法**：熟考型の `Qwen3.5:9B` から、コード生成特化でキレの良い軽量モデル **`qwen2.5-coder:7b`**（または超軽量な1.5B）へ変更する。
- **効果**：パラメータ数が減ることで、M4 Proの広いメモリ帯域幅（273 GB/s）がフルに活き、思考時間が物理的にほぼゼロ（ボタンを押した瞬間に文字が溢れ出す状態）になる。

## 対策B：正しい情報収集を可能にし、嘘を100%排除する ―― RAG実装ルート

- **手法**：ロードマップ3ヶ月目にある **RAG（検索拡張生成）** の簡易版をアプリに組み込む。
- **具体策**：アプリと同じフォルダ内に `npb_info.txt` などの正確なテキストファイルを用意し、AIが回答を組み立てる前にそのファイルを**強制参照（カンニング）**させる機能をコードに数行追加する。
- **効果**：ローカルAIの知識不足を補い、ネット不要で完璧に正確な情報収集ツールへ進化させられる。

# 第3章：M4 Pro パワー検証 & プチRAG実装成功

## 3-1. 「タメはあるが、出力は速い」の正体

### 出力が爆速だった理由（M4 Proの覚醒）

- 1文字目が出た後の「パラパラパラ」という出力速度が圧倒的だったのは、モデルを `7B` クラス（`qwen2.5vl:7b`）に軽量化したため。
- これにより、M4 Proの最大の強みである**広いメモリ帯域幅（273 GB/s）**がフルに活かされた。
- 膨大な脳（モデル）のデータをプロセッサへ一瞬で送り込めるM4 Proの圧倒的なハードウェアパワーが、物理的に証明された瞬間。

### まだ「タメ（最初の沈黙）」があった理由（プチRAGの代償）

- アプリの裏側で「ユーザーが質問 → アプリが `npb_data.txt` を自動読込 → プロンプト（指示文）に合体させてOllamaに投げる」というRAGの基礎処理を挟んだため。
- AIは単なる質問だけでなく、裏で合体された長いカンニングペーパー（知識データ）を一度すべて精密に計算してから1文字目を出力する。そのため初速に数秒の「思考のタメ」が発生する。
- **結論**：精度を高めるためのRAG（外部データ参照）や、論理的思考を求める高度なプロンプトを組み込むほど、計算量に依存して初速のタメは増大する。

▶補足（再現用）：ここで登場するファイル名が `npb_info.txt`（第2章）と `npb_data.txt`（第3章）で揺れている。実装時はどちらか一方に統一すること。速度と精度はトレードオフ（軽量モデルで初速↑ ／ RAG追加で初速↓・精度↑）で、アプリの目的に応じてどちらを優先するか決めるのが設計の勘所。

# 第4章：Zedエディタ 完全ガイド

## 4-1. Zed（ゼッド）とは：次世代AI特化型エディタの正体

Zedは、伝説的エディタ「Atom」や「VS Code」の開発者たちが、ゼロからRust言語で書き直した近代最高峰の超高速コードエディタ。

### 自走力を引き出す3大アドバンテージ

1. **異次元の起動・動作速度**：Rust製で、MacのGPUをフル活用して描画。ファイル読み込み・スクロール・コード補完が一切引っかからず「打てば響く弾丸スピード」で動く。
2. **AIチャット（Assistant Panel）のネイティブ内蔵**：外部プラグイン不要で、画面横にAIとの対話スペースを常駐。コード選択からの自動修正やデバッグ効率が劇的に向上。
3. **ローカルOllamaとのシームレスな合体**：インターネットを遮断した完全オフライン状態のまま、Mac mini（M4 Pro）内部のAI（Qwen等）をエディタの脳として直結できる。

## 4-2. 【永久無料・完全ローカル】環境構築の全手順

Mac mini（M4 Pro / 24GB）のパワーをフル解放し、1円も課金されず永久無料で使うためのセットアップ。

### ステップ①：Zedの導入

```bash
brew install --cask zed
```

インストール後、開発対象のプロジェクトフォルダへ移動し、Zedを起動。

```bash
cd ~/Library/CloudStorage/GoogleDrive-（自分のアカウント）/マイドライブ/local-ai-project
zed .
```

※初回起動時に信頼性確認のポップアップ（Unrecognized Project）が出たら、チェックを入れて「Trust and Continue」を選択。

### ステップ②：Ollama（ローカルAI）との接続設定

1. Zed起動後、`cmd + ,`（カンマ）で設定ファイル（`settings.json`）を開く。
2. 以下のJSONを貼り付け、`cmd + s` で保存。

```json
{
  "theme": "One Dark",
  "assistant": {
    "version": "2",
    "provider": {
      "name": "ollama",
      "api_url": "http://localhost:11434"
    },
    "default_model": "qwen3.5-9b-16k:latest"
  }
}
```

### ステップ③：並列自走開発画面の起動

1. 設定を保存すると、テーマが自動で黒基調の「One Dark」に切り替わる。
2. `cmd + ctrl + a` を同時押し、またはメニューバーから「View → Toggle Assistant Panel」をクリック。
3. 画面左側に縦長のAIチャットスペースが出現。
4. **【スマート同期機能】**：チャット画面のメニューからモデルを別のもの（`qwen2.5vl:7b` など）に切り替えると、裏側の `settings.json` も自動的に最新へリアルタイム更新される。

▶補足（再現用）：`default_model` には `ollama list` に実在するモデル名を正確に入れること。`api_url` の `11434` はOllamaの標準ポート。Zedの設定キー名（`assistant` / `version` 等）はZedのバージョンで変わることがあるため、うまく繋がらない場合は起動中のZedのバージョンに対応した設定キーを確認する。

## 4-3. 有料プラン（Zed Pro）と無料ローカル環境の決定的な違い
